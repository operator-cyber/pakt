<decisions_made>

- writeUserProfile uses get() then conditional update/set instead of set({merge:true}) — Firestore rules block client writes to tier field, and merge:true includes tier in affectedKeys even when value is unchanged
- atLimit early return placed AFTER all React hooks — original placement before useMemo hooks violated rules of hooks, caused React error #300 crash when monthlyUsage changed from 0 to 2 on async Firestore query return
- Cloud upload failure doesn't block local vault save (try/catch around uploadNdaToCloud)
- Usage counting is query-based (createdAt >= firstOfMonth), not a stored counter — auto-resets, tamper-resistant
- Firestore security rules deployed via Console paste, not Firebase CLI

## Critical Anti-Patterns (do NOT repeat these)

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Early return before hooks | Placing conditional returns before useMemo/useEffect/useCallback calls in React components. Causes hooks to be called conditionally, violating React rules. | blocking | Always place early returns AFTER all hook calls in the component. Search for useMemo/useEffect/useCallback after any early return. |
| set({merge:true}) with protected fields | Using Firestore set with merge:true when security rules protect specific fields. affectedKeys() includes ALL keys in the write payload regardless of value change. | advisory | For documents with protected fields, use get() then conditional update/set. Only include mutable fields in update(). |
